# Desktop-to-Web Individual route parity audit

Date: 2026-07-20  
Scope: source-level audit of the authenticated Individual workspace only. Manager Access and backend/data-boundary differences are intentionally excluded from parity defects. No rendered screenshot claim is made by this audit.

## Baseline and interpretation

Desktop routing is defined by `apps/desktop/src/components/shell/ScreenRouter.tsx`, `apps/desktop/src/components/shell/AppShell.tsx`, and `apps/desktop/src/lib/ui.ts`. Web routing is defined by `apps/web/components/IndividualWorkspaceShell.tsx`, `apps/web/lib/individualWorkspaceRoute.ts`, and the composition in `apps/web/app/dashboard/page.tsx`.

The Web allowlist already covers 12 of the Desktop's 13 large-window screens: Today; Capacity, Forecast, Review, AI Usage, Summary; Ask, Accelerate, Skills; Activity, Audit; and Settings. Desktop's conditional `sensitive` / Flagged Captures screen is the only screen without a Web route. That omission must not be solved by uploading raw captures: Web should expose, at most, a desktop-shaped unavailable boundary because `apps/web/components/IndividualHistorySettings.tsx` correctly states that the capture and sensitive summary never reach Web.

The shell is structurally close: the five primary destinations, nested tab order, sidebar metric, responsive collapse, compact-window path, query-string routing, browser history, and roving tab focus all have Web equivalents. Remaining shell differences are visual/interaction contracts, not missing navigation infrastructure: Desktop shows the active week beside Week tabs, provides keyboard shortcut metadata, and uses the native toolbar's tracking/theme/window controls; Web instead uses an API/account toolbar and does not render active-week context. Manager Access remains a separate, permission-gated entry and is not a parity target.

## Recommended implementation order

1. **Agent (`agent`)** — highest product impact and already operational. Consolidating onto Desktop geometry plus copy/retry/follow-up/Mac-action-card states yields the largest visible parity gain without changing the review-safe backend contract.
2. **Forecast + Summary (`forecast`, `narrative`)** — highest Week decision value. Both have enough review-safe data to fill Desktop-shaped deterministic panels; missing AI/local state can be represented honestly in-place.
3. **AI Usage + Accelerate + Skills (`usage`, `accelerate`, `skills`)** — largest raw visual gap. Treat these as a single local-only state-system pass so the pages use authentic Desktop empty/result footprints instead of three bespoke marketing-style boundary cards.
4. **History completeness (`sensitive`, then `audit`/`ledger` polish)** — add truthful deep-link behavior and then converge toolbar/card density.
5. **Shell/Settings/Today/Week/Review screenshot polish** — these are structurally closest; finish spacing, active-week context, responsive behavior, focus, and dark theme after the larger page silhouettes converge.

## Non-negotiable verification matrix

- Use the same synthetic fixture/week for Desktop and authenticated Web screenshots. Capture each route at 1440x900 and 1024x720; capture light/dark for all P0 routes.
- Compare the content column left edge, maximum width, context-tab baseline, header top offset, card grid, typography, border radius, and control height. A route is not pixel-level complete when only text assertions pass.
- Exercise real Web states: no replica, valid replica, integrity/load error, narrow viewport, keyboard-only tab navigation, browser Back/Forward, and action pending/error/success where supported.
- Keep Manager Access out of screenshot diffs except for the permission-gated sidebar entry. Never move Manager data or controls into Individual pages.
- Keep the backend contract unchanged: Web consumes API/Supabase review-safe fields, keeps chat/browser state temporary, and never receives raw window titles, screenshots, notes, local audit events, AI credentials, acceleration recipes, or fabricated usage/forecast values.
- Standing source gates should include `npm run build` and the relevant `apps/web/lib/individual*Parity.test.ts` / presentation tests. Rendered screenshots remain a separate acceptance surface from unit/build success.

## Audit conclusion

Navigation parity is nearly complete, but full Individual experience parity is not. Capacity, Today, Weekly Review, History Activity, and Settings have the strongest foundations. Agent is the best next high-impact finish because it is already useful end-to-end; Forecast/Summary follow. AI Usage, Accelerate, and Skills are the clearest remaining visual outliers. The missing `sensitive` deep-link needs an explicit privacy-safe result rather than silently falling back to Capacity.
