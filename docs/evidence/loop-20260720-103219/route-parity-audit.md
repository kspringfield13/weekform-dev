# Desktop-to-Web Individual route parity audit

Date: 2026-07-20  
Scope: source-level audit of the authenticated Individual workspace only. Manager Access and backend/data-boundary differences are intentionally excluded from parity defects. No rendered screenshot claim is made by this audit.

## Baseline and interpretation

Desktop routing is defined by `apps/desktop/src/components/shell/ScreenRouter.tsx`, `apps/desktop/src/components/shell/AppShell.tsx`, and `apps/desktop/src/lib/ui.ts`. Web routing is defined by `apps/web/components/IndividualWorkspaceShell.tsx`, `apps/web/lib/individualWorkspaceRoute.ts`, and the composition in `apps/web/app/dashboard/page.tsx`.

The Web allowlist now covers all 13 Desktop large-window screens: Today; Capacity, Forecast, Review, AI Usage, Summary; Ask, Accelerate, Skills; Activity, Audit, Flagged Captures; and Settings. Iteration 8 closed the final route-completeness gap by mapping Desktop's conditional `sensitive` screen to a Desktop-shaped unavailable boundary. It does not upload or infer raw captures, summaries, screenshots, local counts, or destructive actions.

The shell is structurally close: the five primary destinations, nested tab order, sidebar metric, responsive collapse, compact-window path, query-string routing, browser history, and roving tab focus all have Web equivalents. Remaining shell differences are visual/interaction contracts, not missing navigation infrastructure: Desktop shows the active week beside Week tabs, provides keyboard shortcut metadata, and uses the native toolbar's tracking/theme/window controls; Web instead uses an API/account toolbar and does not render active-week context. Manager Access remains a separate, permission-gated entry and is not a parity target.

Iteration 1 of the continuation audit also inspected route-level states that sit above every Individual page. `dashboard/loading.tsx` and `dashboard/error.tsx` were the highest-impact layout discontinuity: both discarded the authenticated app geometry for a generic marketing container. They now share `IndividualDashboardBoundaryShell`, which retains the toolbar, five-destination sidebar, Week context strip, 1200 px content frame, responsive collapse, dark-theme tokens, reduced-motion loading behavior, and an operational error retry. The boundary is deliberately data-free: it does not query or infer identity, workload values, teams, or Manager Access while those facts are unresolved.

## Recommended implementation order

1. **Rendered proof for the shared loading/error shell and P0 routes** — the global layout discontinuity is source-closed, but real wide/narrow and light/dark screenshots remain required before pixel-parity approval.
2. **History rendered proof (`audit`/`ledger`/`sensitive` screenshots)** — route completeness and bounded source-level density are closed. Capture the review-safe Activity/Audit states and privacy-safe Flagged boundary beside their local Desktop references.
3. **Shell/Settings/Today/Week/Review rendered proof** — Settings source-level density is now closed. Capture its six tabs, then finish shell active-week context plus Today/Week/Review spacing, responsive behavior, focus, and dark-theme comparison.
4. **Summary follow-through only if the replica contract expands** — generation, model metadata, editable manager copy, and share actions stay Mac-owned; do not reopen the completed deterministic composition merely to imitate unsupported controls.

## Non-negotiable verification matrix

- Use the same synthetic fixture/week for Desktop and authenticated Web screenshots. Capture each route at 1440x900 and 1024x720; capture light/dark for all P0 routes.
- Compare the content column left edge, maximum width, context-tab baseline, header top offset, card grid, typography, border radius, and control height. A route is not pixel-level complete when only text assertions pass.
- Exercise real Web states: no replica, valid replica, integrity/load error, narrow viewport, keyboard-only tab navigation, browser Back/Forward, and action pending/error/success where supported.
- Keep Manager Access out of screenshot diffs except for the permission-gated sidebar entry. Never move Manager data or controls into Individual pages.
- Keep the backend contract unchanged: Web consumes API/Supabase review-safe fields, keeps chat/browser state temporary, and never receives raw window titles, screenshots, notes, local audit events, AI credentials, acceleration recipes, or fabricated usage/forecast values.
- Standing source gates should include `npm run build` and the relevant `apps/web/lib/individual*Parity.test.ts` / presentation tests. Rendered screenshots remain a separate acceptance surface from unit/build success.

## Audit conclusion

Navigation parity is source-complete across all 13 Desktop screens, but full Individual experience parity is not. Capacity, Today, Weekly Review, History Activity, and Settings have the strongest foundations. Forecast and Summary now join Agent Ask, AI Usage's unavailable state, Accelerate's review-safe boundary, Skills' local-library boundary, and the privacy-safe Flagged route as completed bounded source-level parity slices. None of those source checks substitutes for authenticated matched screenshots; History density and route-wide rendered comparison remain the highest-value visual gaps.
